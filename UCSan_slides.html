Good morning everyone. I'm Mingjun Yin from UC Riverside, and today I'll present UCSan, a compilation-based under-constrained execution engine. This is joint work with Zhaorui Li, Ju Chen, Haochen Zeng, and my advisor Chengyu Song. In one sentence: UCSan lets you take an arbitrary set of C/C++ functions out of a large system — like the Linux kernel — compile them into a normal, self-contained executable, and run precise dynamic analysis on them at native speed, with no test harness and no manual changes.
Motivation

Finding bugs in large systems is hard

Our idea — UCSan: under-constrained execution as a compiler pass

  • An LLVM pass + runtime library: compiles any set of C/C++ functions into a self-contained executable → native speed
  • Decoupled from the dynamic analysis — combine with concolic execution, fuzzing, or model checking
UCSan · USENIX OSDI 20263
The ideal would be: extract a code subset, make it self-contained, and point precise dynamic analysis at it. People have done this manually — extracted TCP stacks, the Linux Kernel Library — but manual extraction is labor-intensive and unmaintainable. UC-KLEE pioneered the automated version: under-constrained execution with lazy initialization — when a pointer is dereferenced for the first time and points to nothing, allocate the object on the fly. The gap: every existing under-constrained engine is interpreter-based. KLEE is roughly 3,000 times slower than native; Angr over 321,000 times — not enough for a file system, the TCP stack, or a driver. Our idea is simple to state: do under-constrained execution as a compiler pass. UCSan is an LLVM pass plus a runtime library built on the sanitizer framework; it compiles an arbitrary set of functions into a self-contained executable that runs at native speed, and it is decoupled from whatever analysis sits on top. To our knowledge, the first compilation-based engine for under-constrained execution. Let me show exactly what that means with a running example.

Compile-time transformation

What UCSan compiles in

Generated entry point — main()

A shadow pointer per pointer argument, deserialize the seed into head, then call cal() → a self-contained binary.

Pointer translation — check_ptr()

Every memory access is translated before use. Aliased pointers (curr, curr->v, head->next) share one shadow pointer — like a segment register. Translated pointers are ephemeral: the object may be reallocated or resized later.

External functions

Map to a wrapper (kmalloc → malloc) · assume pure · or havoc pointer args. (None in this example.)

This is what UCSan compiles in — shown at source level for readability; the real transformation happens on LLVM IR. Green lines are inserted; struck-out lines are replaced. Three parts. First, a generated main(): shadow pointer per pointer argument, deserialize the seed into head, call cal(). Third, external functions: wrapper mapping like kmalloc→malloc, assume pure, or havoc pointer arguments. This example has none.
Two ideas — thirty seconds

JITI & pseudo- vs. real pointers

Watch for these in the next three runs: every pointer the program touches is a pseudo-pointer · check_ptr turns it into a real pointer just in time · JITI materializes memory the first time it's needed.

UCSan · USENIX OSDI 20266
Before we run it, two pieces of vocabulary — thirty seconds. After the transformation, the pointer values the program manipulates, like head equals 0x1, are pseudo-pointers: logical addresses that don't address real memory. Only the shadow pointer knows the actual backing object — it works exactly like a segment register, and all aliased pointers share one. When the program is about to dereference a pseudo-pointer for the first time, UCSan performs Just-In-Time Initialization, JITI: it allocates the object on the fly as a phantom, zero-initialized. check_ptr then hands back a real pointer — an actual usable address inside that object. Real pointers are deliberately ephemeral: used for one access and thrown away, because the backing object may later be reallocated or resized. Keep these three in mind — you'll see all of them in action across the next three seeds.
Plug in an analysis

Exploring paths needs a solver — so plug one in

The binary runs — but exploring paths needs new seeds. UCSan is decoupled, so that’s just composition.

= UCSan† — a complete under-constrained concolic engine; this loop drives the exploration you’re about to watch.

UCSan · USENIX OSDI 20267
At this point we have a self-contained binary that runs. To explore different paths, something has to generate new seeds. Because UCSan is decoupled, that’s just composition: we plug in SymSan — a compilation-based concolic engine that traces symbolic constraints and solves them with an SMT solver — plus Thoroupy, a small Python companion that orchestrates exploration. The combination is UCSan-dagger, and it plays the role of “the solver” in the walkthrough you’re about to see.
Concolic exploration ① ② ③

Seed 0 — start with an empty seed

①②get_concrete(head): not in seed → phantom object, zero-filled → head = 0x0

·while(head): 0 → loop skipped, sum = 0 → assert NOT triggered

③Solver negates head ≠ null → assign head = 0x1 (a pseudo-pointer — any non-null value works) ⟹ Seed 1

UCSan · USENIX OSDI 20268
Let's execute. Seed 0 is completely empty. get_concrete looks up head in the seed, finds nothing, and creates a phantom: it pretends the object exists and zero-fills it. head becomes null, the loop is skipped, sum stays zero, no assert. Path one done. Now the concolic layer negates the branch — head must be non-null — and generates Seed 1 with head = 0x1. It does not need a valid address, just "not null". These program-level pointer values are pseudo-pointers: they say something is pointed to; the shadow pointer decides where it actually lives.
Concolic exploration ④ ⑤ ⑥ ⑦

Seed 1 — JIT-initialize on first access

⑤check_ptr → JITI allocates obj1 (phantom): size from container_of → node_t, not list_t · pointed-by SO@0 recorded

⑥pseudo-pointer head->next (0x1) → real obj1.list.next; shadow ptr stores logical base −3 (1−4), so any pointer arithmetic keeps working

⑦obj1.next = 0 → loop exits · sum = 0+obj1.v is symbolic → solve sum>100 ∧ obj1.next≠0 ⟹ Seed 2

UCSan · USENIX OSDI 20269
With Seed 1, head is 0x1, we enter the loop, and the first dereference triggers Just-In-Time Initialization. check_ptr allocates obj1 as a phantom. Three challenges right here. One: size — trusting head's declared type list_t would under-allocate; UCSan instruments pointer arithmetic and casts, so container_of reveals the real type, node_t. Two: offset — the pointer lands on the embedded next field, and the shadow pointer records the logical base, minus three, because container_of subtracted four from 0x1 — v is recovered through that negative offset. Three: future seeds — UCSan records that obj1 is pointed-by slot 0 of the super object. obj1 is zero-filled, so next is null and the loop exits. But sum is now symbolic — zero plus obj1.v. The solver asks for sum greater than 100 and obj1.next non-null: Seed 2. Note obj1.next can also be 0x1, the same value as head — same logical address in different segments never conflicts.
Concolic exploration ⑧ ⑨ ⑩

Seed 2 — replay objects & trigger the bug

⑨obj1 loaded from seed (via pointed-by recorded at ⑤) → obj1.next = 0x1 → re-enter loop

⑩check_ptr → JITI obj2 (phantom); record point-to (obj1 off 4 → obj2); obj2.next = 0 ends the loop

sum = 101 → assert(false) FIRES ✓   all paths covered — no harness, no manual setup

UCSan · USENIX OSDI 202610
Seed 2 contains two serialized objects: the super object and obj1, with v = 101 and next = 0x1. This time, when check_ptr materializes obj1, it finds the content in the seed via the pointed-by relation recorded earlier — that's what makes execution deterministically reproducible. obj1.next is 0x1, so we re-enter the loop, and JITI allocates a second phantom, obj2, recording that obj1's next field points to it. obj2 is zero-filled, so the loop ends. sum is 101 — the assertion fires. Three seeds, all paths, and nobody wrote a harness or built a linked list by hand.

Config is the harness-killer: name any function as entry, any set of functions as scope. Write it by hand — or generate it with static analysis or an LLM.

Output runs anywhere: no VM, no disk image, no device. Kernel code becomes a normal user-space process.

UCSan · USENIX OSDI 202611
Stepping back: everything in that walkthrough is produced by this fully automatic pipeline. Two inputs: the program as LLVM IR — any front end that emits IR works; for the kernel, enabling LTO gets you the IR — and a small YAML config naming the entry function, the analysis scope, and how to handle out-of-scope functions. The config can be written by hand, or generated by static analysis or an LLM. UCSan instruments every memory access for address translation, stubs or wraps external calls, and drops out-of-scope code. Any analysis pass slots into the pipeline the same way SymSan did — fuzzing or model checking work too. And the output links into a normal user-space executable: no VM, no disk image, no device. That generality is what we now evaluate.

Second case study: reproducing known vulnerabilities with no hand-written harness — just a scope and an entry function, with most configurations generated automatically, including by a coding agent. Across three datasets: all 30 NVD CVEs we sampled were reproduced; 69 of 94 CVEs in the AFGen dataset re-discovered; and 26 of the 38 kernel bugs from the SyzSpec paper — without a fuzzer and without syscall descriptions. The misses are mostly path explosion inside large scopes or scopes that didn't include the buggy code — not fundamental limitations.
